# ahb_arbiter_2m Design Spec

## 1. Scope

`ahb_arbiter_2m` arbitrates between two AHB-Lite masters:

```text
m0 = core
m1 = dma
```

It forwards one selected master's address/control/write-data channel to the
shared AHB fabric and routes the shared response back to the same master. The
arbiter is deliberately non-pipelined: it completes one single-beat transfer
before granting the next one.

| `m0_*` outputs | output | Core master read data/ready/response |
| `m1_*` inputs | input | DMA master address/control/write data |
| `m1_*` outputs | output | DMA master read data/ready/response |
| shared AHB outputs | output | Selected address/control/write data |
| shared AHB inputs | input | Response from slave mux |
| `grant_valid_o` | output | High during the emitted address phase |
| `grant_idx_o` | output | Address-phase granted master, 0 for core, 1 for DMA |

## 4. Behavior

Request detection:

```text
master requests when HTRANS[1] = 1
NONSEQ and SEQ are treated as active requests
IDLE and BUSY are not treated as requests
```

Grant policy:

```text
reset:
  no grant
  last_grant_q = m1, so first simultaneous request grants m0

m0 only:
  grant m0

m1 only:
  grant m1

m0 and m1:
  grant the master that did not win the previous accepted grant
```

Transaction policy:

```text
ARB_IDLE:
  choose m0/m1 using round-robin tie break

ARB_ADDR:
  drive one NONSEQ address/control phase
  selected master sees HREADY=1 for address acceptance

ARB_WAIT:
  drive HTRANS=IDLE
  hold HWDATA for writes while the registered slave produces response
  selected master sees HREADY=0
  if slave HREADY=1 -> ARB_RESP

ARB_RESP:
  route HRDATA/HRESP/HREADY to the selected owner
  selected master samples the response
  if slave HREADY=1 -> ARB_IDLE
```

Response routing:

```text
only the outstanding transaction owner receives HRDATA/HREADY/HRESP
non-granted requesting master sees HREADY low
idle non-granted master sees HREADY high
non-granted master HRDATA is zero and HRESP is OKAY
```

## 5. Grant State Diagram

![ahb_arbiter_2m grant state](images/ahb_arbiter_2m_state.png)

PNG generated by `docs/tools/render_state_pngs.py`.

The arbiter uses the named FSM `ARB_IDLE`, `ARB_ADDR`, `ARB_WAIT`, and
`ARB_RESP`. `owner_q` is the response-route owner for the outstanding transfer.

```text
Reset:
  state_q = ARB_IDLE
  owner_q = m0
  last_grant_q = m1

ARB_IDLE:
  no request -> stay ARB_IDLE
  only m0 -> owner_q=m0, last_grant_q=m0, ARB_ADDR
  only m1 -> owner_q=m1, last_grant_q=m1, ARB_ADDR
  both -> choose opposite last_grant_q, update last_grant_q, ARB_ADDR

ARB_ADDR:
  one-cycle address phase -> ARB_WAIT

ARB_WAIT:
  HREADY=0 -> stay ARB_WAIT, HWDATA held
  HREADY=1 -> ARB_RESP

ARB_RESP:
  HREADY=0 -> stay ARB_RESP, response held to owner
  HREADY=1 -> ARB_IDLE
```

This serializes the bus so the address-phase owner and response-phase owner
cannot diverge.

## 6. Verification Summary

Verified by `tb_ahb_arbiter_2m`.

Coverage includes:

```text
reset no-grant state
m0-only grant
m1-only grant
simultaneous request round-robin alternation
WAIT-state HREADY low hold
selected master response routing
non-selected requesting master HREADY low
response ERROR routing
write-data hold through the registered slave data phase
```
